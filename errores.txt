Get-Process -Name python 2>$null | Stop-Process -Force
Eso mata TODOS los procesos python... incluído el proceso del propio agente. Por eso ves "Tool execution aborted" y "interrupted" en la línea 77-78.
El ciclo es:
1. Agente encuentra bug (ultimo_costo → costo_promedio)
2. Lo arregla en código ✅
3. Mata todos los python para reiniciar → se mata a sí mismo 💀
4. El humano escribe "te quedaste trabado?"
5. El agente revive, pero el API también se reinició
6. Prueba el endpoint y sigue fallando porque el código nuevo no se aplicó bien (el API nueva arrancó antes de que el fix se guardara, o es la instancia del health check automático)
7. Encuentra otro bug (COALESCE), lo arregla, reinicia → PID 9000
8. Y ahí se corta errores.txt — no sabemos si llegó a probar
No va a salir solo porque su mecanismo de restart (Stop-Process -Force sobre -Name python) siempre lo va a matar a él también. Es un deadlock: necesita reiniciar la API para aplicar los fixes, pero reiniciar así lo mata a él.
Solución: decile que NO use Stop-Process -Force sobre todos los python. Que busque SOLO el proceso de uvicorn por el puerto 8000, o que le pase el PID específico, o que directamente edite el código y le pida al humano/scheduled task que reinicie la API.

---

Regla aprendida: para reiniciar la API sin matar al agente, buscar proceso por puerto 8000:
```
$pidApi = (netstat -ano | Select-String ":8000.*LISTENING" | ForEach-Object { $_ -split '\s+' | Select-Object -Last 1 }); if ($pidApi) { Stop-Process -Id $pidApi -Force }
```
O directamente pedirle al humano que ejecute `.\infra\start_api.ps1`.
